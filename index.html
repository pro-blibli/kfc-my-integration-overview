How Talon.One Works

The Incentive Platform Engine

Talon.One works as a real-time incentive evaluation engine. KFC Malaysia's Middleware sends customer and order context via API — Talon.One evaluates all active campaigns against that context — and returns a list of incentive instructions called effects. All promotional logic lives in T1, not in MW code.

01

MW sends customer context and order data to Talon.One via the Integration API on every cart update and checkout.

Session State — the Commit Switch

open
Rules evaluate on every update. Effects returned but budgets and points not yet committed. Use to show discount previews before payment.
closed
Payment confirmed. Coupons redeemed, points awarded, budget counters decremented. Only send after successful payment.
cancelled
Full rollback. Points reversed, voucher reinstated (usageCount −1), budgets reverted. Send on payment failure or full refund.
partially_returned
Per-item rollback. Used when a specific item (e.g. out-of-stock drink) is unavailable. Remaining items still count toward budgets.

Effects
Incentive instructions
Effects are the output of the T1 Rule Engine. Returned synchronously in every session or event API response, they tell MW exactly what to do — apply a discount, award points, add a free item, reject a coupon. MW must parse and act on every effect type.
setDiscountDiscount on session total or per-item — sum ALL instances
addLoyaltyPointsAward pts — store transactionUUID for reversal
acceptCoupon / rejectCouponCoupon validation result + rejectionReason
addFreeItemAdd zero-cost SKU to order (e.g. Gold free drink)
trigger-webhookT1 fires webhook to Mapp / external system
showNotificationDisplay message / banner to customer in App
rollbackCouponCoupon reinstated on session cancellation
Effects live inside
🛒 Customer Session → effects[ ] in API response
⚡ Event → effects[ ] in API response (silent=no)
Key rule
HTTP 200 ≠ promotion applied. Always iterate effects[] and check effectType before acting. Empty effects[] = no campaign conditions matched.
Attributes

Built-in vs Custom Attributes

Attributes are the data fields attached to Talon.One entities. They're what make the Rule Engine context-aware. By passing the right attributes on every session, you enable rules like: "If customer tier is Gold AND channel is App AND cart total exceeds RM50 → apply 20% discount."

📦
Built-in Attributes
Provided by Talon.One · No setup needed
Pre-defined on every entity. Cover the core data model — cart item prices, session totals, coupon codes, profileId. Available immediately, no Campaign Manager configuration required.

Defined by KFC MY · Must register first
Created in Campaign Manager → Account → Attributes, then populated by MW via API. Once registered, the Rule Engine can reference them in any campaign condition or webhook payload.

Customisable Webhooks

Webhooks — Push Events to External Systems

Webhooks in Talon.One are always triggered as a rule effect — they cannot fire independently. Use them to push loyalty events, tier changes, and order data from T1 to Mapp CRM, Treasure Data, or any external endpoint in real time.

STEP 01 📢
Rule fires in T1
A campaign rule condition is met — e.g. customer crosses Gold tier threshold. The rule includes a trigger-webhook effect.
→
STEP 02 📡
T1 sends HTTP request
T1 sends a POST (or PUT/GET/PATCH) to your registered endpoint URL with a payload built from attribute references resolved at fire-time.
→
STEP 03 ✅
Endpoint responds HTTP 2xx
Your endpoint processes the payload and returns 2xx. T1 logs success. Store the X-UUID header as your idempotency key to handle retries.
→
STEP 04 🔔
Downstream system updated
Mapp triggers push + email campaign. Treasure Data receives balance update. Your downstream system acts on the real-time event data.
Webhook payload template (defined per webhook)
// Attribute references are resolved at fire-time
// Built-in: ${$Entity.AttributeName}
// Custom:   ${$Entity.Attributes.X}
// Entities: Profile · Session · Coupon · Event

{
  "customer_id":  "${$Profile.IntegrationId}",
  "session_id":   "${$Session.IntegrationId}",
  "orderTotal":   "${$Session.Total}",
  "loyaltyTier":  "${$Profile.Attributes.loyaltyTier}",
  "channel":      "${$Session.Attributes.channel}",
  "storeId":      "${$Session.Attributes.storeId}",
  "event_type":   "${$bodyText}",  // rule-supplied
  "new_tier":     "Gold"           // hardcoded in rule
}

// Default headers sent by T1:
// Content-Type: application/json
// X-UUID: <unique per request> ← store as idempotency key
Configuration options
HTTP Methods
GET · POST · PUT · PATCH · DELETE — configured per webhook. Custom headers and stored credentials supported.
Attribute reference syntax
Built-in: ${$Entity.AttributeName}
Custom: ${$Entity.Attributes.X}
Exact casing required for entity names.
Idempotency
T1 may retry on non-2xx. Store X-UUID header per delivery. Use it as your idempotency key to handle duplicate deliveries gracefully.
Trigger
Always a rule effect — webhooks cannot fire independently of rule evaluation. Configured in Campaign Manager per webhook.

Post-session-close, MW queries Management API for current balance and pushes to Treasure Data via TD Ingestion API for segmentation and analytics.
→ Treasure Data

Mapp detects birthday → calls MW webhook → MW calls Management API to create a unique birthday voucher for the customer's wallet. App shows voucher on next open.
→ MW → T1 Mgmt API

T1 rule detects customer lapsed >90 days on next purchase. Webhook fires to Mapp with lapsed customer flag. Mapp triggers "Welcome back" campaign with goodwill offer.
→ Mapp CRM
💡
Data retention: T1 API logs and session history have a ~30-day rolling retention window. Stream webhook events to S3 from day one — retroactive export is not possible once logs expire. Use the webhook → S3 → Athena → QuickSight pipeline for long-term loyalty and promotions analytics.
